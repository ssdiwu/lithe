# 经验笔记

## WKWebView 中的中文输入法标点延迟

### 现象

中文输入法开启时，在 xterm 终端连续输入标点，前几个字符可能没有立刻
出现，而是在后续按键时一次性提交。普通中文文字输入不一定受影响。

### 做法

不要在应用快捷键层接管真实的 IME composition 事件。对于没有发生真实
composition 的标点输入，在 window capture 阶段、xterm 处理 `keydown`
之前记录隐藏 textarea 的基线。如果 xterm 已提交英文按键，`onData` 会清除
基线；否则在下一次 `keyup` 比较并补发差异。不要依赖固定键码：WebKit
可能使用 229/0，第三方 macOS 输入法也可能保留逗号等物理键码。补发后仅
抑制一次可能迟到的重复数据。

### 证据

- Lithe 的回归测试覆盖 `keyup=0` 的中文标点、`keyup=32` 的输入法空格、
  第三方输入法保留物理标点键码、多个字符合并延迟、等长全角替换、原生
  xterm 已提交数据以及真实 composition 序列。
- xterm.js 上游 PR #5836 记录了相同的 WebKit 事件时序，并采用 keyup
  fallback 修复。

## 本地化浮层不能假设英文固定宽度

### 现象

终端选中文本后出现的 AI 快捷操作按英文文案固定为 `110px`。切换简体中文
后，文案和快捷键总宽度超过容器，按钮文字被折成两行。

### 做法

让浮层按实际本地化内容取得宽度，再用测得宽度计算居中和视口边界。按钮
文本保持单行；极窄窗口只允许 label 截断，快捷键不缩小。

### 证据

- `SelectionAskAi.test.ts` 覆盖可变 popup 宽度的居中与左右边界。
- `SelectionAskAi.tsx` 使用元素实际宽度定位，不再把英文宽度同时当作布局
  和定位常量。
